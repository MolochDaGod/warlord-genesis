01The trick: the server isn't separate

Most teams stand up a whole second machine for real-time play — a socket box, its own ports, its own deploy pipeline. Carrier doesn't. The PvP room is a WebSocket upgrade bolted onto the same Express HTTP server that already serves the API.

One server, two doors. Normal traffic stays HTTP. When a browser asks to "upgrade" the connection on the path /api/carrier, the same process hands it to the game room instead. Same deploy, same domain, same TLS — no extra moving parts.

artifacts/api-server/src/game/carrier-server.ts

// The PvP room attaches to the EXISTING http server — no new process.
const wss = new WebSocketServer({ noServer: true });
const room = getCarrierRoom();        // one shared room, every commander

server.on("upgrade", (req, socket, head) => {
  if (pathname !== "/api/carrier") return;  // only OUR path upgrades
  wss.handleUpgrade(req, socket, head, (ws) => wss.emit("connection", ws, req));
});

03Why the same code runs on both ends

The physics, the ship stepping, the message format — all live in one shared library, @workspace/carrier-net. The client imports it to predict; the server imports it to rule. They can never drift, because there is only one copy.

🧠

One sim, two roles

The exact stepShip the client predicts with is the one the server enforces. No "client version vs server version" divergence to debug.

🎲

Deterministic world

Planets, asteroids, reward boxes and enemy pockets all derive from a seed + tick count — reproducible, fair, and trivially testable.

🔌

Auto protocol

The client picks wss:// on HTTPS and ws:// in dev from window.location.host — zero URLs to hardcode per environment.

🛰️

Isolated room

Carrier owns its /api/carrier path, separate from sibling games — so its state never collides with anyone else's match.

artifacts/carrier/src/game/net.ts — the client never hardcodes a host

function wsUrl() {
  const proto = window.location.protocol === "https:" ? "wss:" : "ws:";
  return `${proto}//${window.location.host}${WS_PATH}`;  // follows wherever it's deployed
}

04Going live, step by step

Because the realtime server is just part of the API server, "deploying PvP" is the same motion as deploying any backend. Here is the whole thing.

Build the shared contract

Generate API hooks + schemas so client and server speak the same language: pnpm --filter @workspace/api-spec run codegen

Provision persistence

Point DATABASE_URL at Postgres and push the schema: pnpm --filter @workspace/db run push. Identity rides on Puter — no auth servers to run.

Bundle the server

One esbuild pass produces the runnable backend (API + WebSocket room together): pnpm run build

Publish

Hit Publish. The platform serves the build over HTTPS and routes /api/carrier to the same process — the WebSocket upgrade just works, TLS included.

Players connect

The Carrier app derives wss://your-domain/api/carrier from the browser URL automatically. First player in spawns the shared room; everyone else joins it. PvP is live.

That's the whole secret

No socket cluster. No second deploy. No port wrangling. The authoritative arena ships with the API server you were already deploying.
